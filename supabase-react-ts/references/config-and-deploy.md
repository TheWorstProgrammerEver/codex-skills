# Config And Deploy

Use this when changing runtime config, local/LAN setup, Netlify builds, Supabase project config, or deployment scripts.

## Runtime Config Files

The starter uses JSON config plus a tiny synchronous browser loader:

- `public/config.js`: committed loader.
- `public/config.json`: committed deployment template with substitution tokens.
- `public/config.local.json`: ignored local machine config generated by `get-going`.
- `tests/visual/config.test.json`: test config routed by Playwright.

`config.js` defaults to `/config.local.json` when `#{CONFIG_FILE}#` has not been substituted. Deployed builds should substitute the config file path so production does not depend on local-only files.

Do not rewrite committed source config for deployment. Mutate built `dist` assets only.

## Local/LAN Workflow

Prefer scripts over one-off command sequences.

`npm run get-going` should:

- install dependencies if missing
- ensure Docker Desktop is running on macOS
- start local Supabase
- serve local Edge Functions
- start Vite with `--host 0.0.0.0`
- generate ignored `public/config.local.json`
- print app, Supabase API, Studio, Mailpit, and health endpoints for localhost and LAN

`npm run all-done` should:

- stop Vite/dev processes
- stop local Edge Functions
- stop Supabase
- print endpoint status

Treat exact `192.168.*` addresses as ephemeral. Scripts should discover them.

## Supabase Local

Keep `supabase/config.toml` compatible with current Supabase CLI. If GitHub/Supabase Preview rejects a config key, remove obsolete local-only defaults rather than cargo-culting old config.

Common local endpoints:

- API: `http://127.0.0.1:54321`
- DB: `postgresql://postgres:postgres@127.0.0.1:54322/postgres`
- Studio: `http://127.0.0.1:54323`
- Mailpit: `http://127.0.0.1:54324`

Local function health endpoints should be cheap and unauthenticated.

## Netlify Build

Use `npm run build:netlify` for Netlify builds.

The build script should:

1. Run the normal app build.
2. Render `dist/config.js` so it loads `/config.json`.
3. Substitute env vars into `dist/config.json`.
4. Parse the rendered JSON to catch invalid output.
5. Leave source files untouched.

Do not hard-code a duplicate list of boolean config keys in the build script. Let JSON parse/runtime validation catch bad values.

## Deployment Variables

Keep app config distinct from Supabase as an implementation detail:

```json
{
  "appName": "Team Tasks",
  "buildVersion": "#{BUILD_VERSION}#",
  "environment": "#{ENVIRONMENT}#",
  "auth": {
    "supportedTypes": {
      "emailPassword": #{AUTH_EMAIL_PASSWORD_ENABLED}#,
      "otp": #{AUTH_OTP_ENABLED}#,
      "magicLink": #{AUTH_MAGIC_LINK_ENABLED}#
    }
  },
  "supabase": {
    "url": "#{SUPABASE_URL}#",
    "publishableKey": "#{SUPABASE_PUBLISHABLE_KEY}#"
  }
}
```

Update names/tokens for the product, but preserve the separation.

## Browser Redirects

For Netlify-hosted SPAs, include:

```text
/* /index.html 200
```

in `public/_redirects`.

## Supabase Hosted Project

Set Auth redirect/site URLs in the Supabase dashboard for deployed domains. Project config in the repo helps local/dev parity but does not replace hosted project dashboard settings for every auth provider.

For Edge Function CORS, prefer Supabase-supported wrappers/platform behavior when available. It is acceptable to serve permissive CORS for authenticated app functions if RLS and auth checks prevent useful abuse.
